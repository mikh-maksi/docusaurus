# Виклад обсягу проєкту (Project Scope Statement)
Виклад обсягу проєкту (Project Scope Statement) — це офіційний документ, який описує цілі, завдання, ключові результати, межі, вимоги та обмеження проєкту. Він слугує основою для роботи команди, запобігає розростанню завдань та узгоджує очікування між замовником і виконавцями.
Якісний виклад обсягу обов'язково включає наступні структуровані елементи:
1. Головні компоненти
Мета та цілі: Конкретні результати, які ви прагнете отримати по завершенню (вони мають відповідати критеріям SMART). 	Очікувані результати (Deliverables): Відчутні продукти, послуги або артефакти, які будуть створені (наприклад, готова програма, розроблений дизайн чи збудований об'єкт). Вимоги до проєкту: Технічні, бізнесові та функціональні критерії, яким мають відповідати кінцеві результати.
2. Межі проєкту (Що входить, а що ні)
    	Включення: Визначення того, які саме роботи вважатимуться виконаними. Виключення: Чіткий перелік дій, функцій або робіт, які не є частиною цього проєкту. Це допомагає уникнути непорозумінь.
3. Обмеження та припущення
Обмеження: Зовнішні чи внутрішні фактори, що обмежують команду, наприклад: фіксований бюджет, кінцевий дедлайн, брак кадрів або законодавчі норми. Припущення: Фактори, які ви вважаєте правдивими, але не можете на 100% контролювати на старті (наприклад: своєчасне затвердження документів клієнтом, стабільність курсу валют тощо).
Навіщо це потрібно? Запобігає «розростанню» (Scope Creep): Захищає від безконтрольного додавання нових завдань та функцій, які не були погоджені спочатку. Фокусує команду: Забезпечує єдине бачення проєкту для стейкхолдерів. Основа для оцінки: Дає можливість точно вирахувати необхідний бюджет і час для складання графіка робіт.
Матриця відстеження вимог (Requirements Traceability Matrix — RTM) — це інструмент (таблиця), який використовується для встановлення, перевірки та відстеження двосторонніх зв'язків між початковими вимогами проєкту та кінцевими артефактами (елементами дизайну, кодом, тест-кейсами), щоб гарантувати повне виконання всіх завдань.
Основні види матриць вимог, їхня структура та призначення:
1. Матриця простежуваності вперед (Forward Traceability). Слугує для того, щоб переконатися, що кожна бізнес-вимога розробляється та втілюється у готовий продукт. Зв'язок: Від Вимог (Requirements) до Дизайну (Design) та коду.
2. Матриця простежуваності назад (Backward / Reverse Traceability). Слугує для гарантії того, що в проєкті не створюється зайвий функціонал, не передбачений початковими бізнес-вимогами. Зв'язок: Від артефактів (Код / Дизайн) назад до початкових Вимог.
 3. Матриця покриття тестами (Test Traceability Matrix) / Матриця відповідності. Використовується для валідації того, що всі вимоги до продукту покриті тестами. Зв'язок: Вимога ➔ Тест-кейс (Test Case) ➔ Результат тесту.
📊 Типова структура матриці
У класичному вигляді це таблиця з такими колонками:  ID Вимоги: Унікальний ідентифікатор (наприклад, REQ-001). Опис вимоги: Суть того, що має бути реалізовано. Джерело вимоги: Звідки надійшла (замовник, документ, користувач). ID Тест-кейса: Номер тесту для перевірки цієї вимоги. Статус: Виконано, провалено, в процесі.
Чому RTM важлива для команди? Повний контроль: Показує, які функції ще не розроблені або не протестовані. Аналіз впливу (Impact Analysis): Допомагає зрозуміти, які модулі системи постраждають або зміняться, якщо замовник вирішить внести зміни в одну з вимог. Прозорість: Доводить стейкхолдерам (замовникам), що робота виконана згідно з технічним завданням.
